# 股票分析器

本技能基于 `src/services/analyzer_service.py` 的逻辑，提供分析股票和整体市场的功能。

## A股交易者版 Report 仪表盘规则（强制）

用户默认只交易 A 股与 A 股 ETF。生成 report、复盘、盘中看盘、明日计划、减仓/加仓建议时，必须优先输出“A股持仓决策仪表盘”，再保留 Markdown 详细原文/归档。报告不得把美股、韩国、日本写成海外交易建议；外围市场只能作为 A 股科技线、半导体、AI链、通信、消费电子的风险偏好雷达。

仪表盘必须包含这些结构：
- `account_status`：总资产、当日盈亏、仓位、现金、风险灯、当前姿态（进攻/均衡/防守/强防守）。缺失就写“未取得”，不能编。
- `cn_market`：上证、深成指、创业板、科创50、成交额、涨跌家数、主力资金、科技/通信/消费电子相关板块强弱。
- `external_radar`：美股（纳指、纳指100、费半/SOX、QQQ、SOXX、VIX）、韩国（KOSPI、KOSDAQ）、日本（日经225、TOPIX）。每项只输出“利多/中性/利空/数据缺失”，并标注数据时间或缺失原因。
- `portfolio_matrix`：每个 A 股持仓一行，包含当前动作、仓位角色、强弱、BBI位置、资金方向、关键价位、明日触发动作。
- `trade_timeline`：按时间排序买卖点，标注是否符合“先试仓、确认后加仓、买强不买弱、去弱留强、做错不加仓”。
- `next_session_plan`：开盘15分钟、10:30、14:30后，分别给触发价/条件、动作、比例、失效条件。

外部市场写法约束：
- 可以写“纳指/费半走弱 -> A股科技线风险偏好承压”。
- 不可以写“买入/卖出 QQQ、SOXX、KOSPI、日经”等海外交易动作。
- 外围数据缺失、滞后或未验证时，必须显示“数据缺失/未取得/滞后”，不得推断成确定结论。

UI/Markdown 输出要求：
- 主视图使用“决策矩阵版”：少长文、多卡片、多标签、多价格线。
- Markdown 继续保留为“详细原文/归档”，但不是主要阅读入口。
- 未操作的持仓不得被写成“今日操作错误”；只能评价实际发生的交易。

## 辅助判断库

当用户要求结合个人经验、技术规则、段永平/巴菲特等长期投资方法论来判断给定标的时，本技能应把 `trading_judgment_library/` 作为辅助规则库使用。

判断库的边界：

- 只处理用户已经给出的标的、持仓或观察对象。
- 不做全市场选股，不生成股票池，不主动推荐新标的。
- 不执行真实下单，不承诺收益。
- 用于回答“能不能做、怎么做、什么时候承认错”，而不是简单预测涨跌。

辅助规则库入口：

- 设计说明：`docs/trading-judgment-library.md`
- 巴菲特原则样例：`trading_judgment_library/philosophy/buffett_principles.yaml`
- 段永平原则样例：`trading_judgment_library/philosophy/duan_principles.yaml`
- 量价技术规则：`trading_judgment_library/technical/volume_price_rules.yaml`
- 硬风控规则：`trading_judgment_library/risk/hard_reject_rules.yaml`
- 仓位与回撤规则：`trading_judgment_library/risk/position_rules.yaml`
- 质量成长回踩框架：`trading_judgment_library/playbooks/quality_growth_pullback.yaml`

使用方式：

1. 先按 `stock_analyzer` 原有流程生成个股或持仓分析。
2. 再用判断库的认知镜片复核：`user_lens`、`buffett_lens`、`duan_lens`、`technical_lens`、`risk_lens`。
3. 输出细分状态：`回避 / 观察 / 等买点 / 试仓 / 持有 / 加仓 / 减仓 / 清仓`。
4. 写入或建议写入 `DecisionSignal` 时，将细分状态映射为现有动作：`avoid / watch / buy / hold / add / reduce / sell`。
5. 在 `evidence` 中保留命中的镜片和规则 ID，例如 `matched_lenses`、`matched_rules`，方便后验评估和复盘。

用户固定交易规则：

- 买强不买弱：同一科技主线、持仓或观察池里，优先考虑相对强势、趋势完整、资金承接好的标的；弱票便宜不等于机会。
- 去弱留强：当持仓或观察池出现明显强弱分化时，降低持续跑输标的的优先级，保留或优先配置强势标的。
- 长期趋势票看 BBI：仅适用于长期趋势向上的票。回踩 BBI 可作为重要支撑和散户割肉线参考，优先看周K确认；若周K有效跌破 BBI、BBI拐头或长期趋势转弱，则该规则失效。

每次生成分析、持仓报告、决策仪或操作建议时，必须执行以下复核：

1. 读取并应用 `trading_judgment_library/technical/volume_price_rules.yaml`、`trading_judgment_library/risk/hard_reject_rules.yaml` 与 `trading_judgment_library/risk/position_rules.yaml`。
2. 在报告中加入“个人规则复核”或等价小节，明确输出是否命中 `买强不买弱`、`去弱留强`、`长期趋势票 BBI 支撑线` 等规则。
3. 对长期趋势向上的标的，必须检查日线 BBI 与周线/周K BBI 附近支撑；如果价格打到 BBI，要按“散户割肉线/趋势支撑观察位”处理，不能直接用短线破位口径误判。
4. 对非长期趋势标的，也要说明 BBI 是否不适用或仅作弱参考，不能把 BBI 机械套到震荡票、题材急涨票或短线交易票上。
5. 最终建议必须同时结合相对强弱、BBI/周K、硬风控规则和仓位状态，不能只凭单一指标给出买卖结论。

当用户说“结合段永平/巴菲特”“按我的规则库判断”“用交易认知判断框架”时，应在 `stock_analyzer` 分析结果基础上，把上述判断库作为参考建议层，而不是切换到独立选股或独立策略 skill。


## 技术面与资金面强制复核

以后生成股票分析、持仓报告、复盘、决策仪或明日操作建议时，必须把“技术分析 + 资金流/盘口资金”纳入判断范围，不能只看涨跌幅、均线或单一 BBI。

必须至少检查并写入结论：

1. 趋势结构：日线/周线趋势、BBI、MA5/MA10/MA20/MA60、均线斜率、价格相对关键均线的位置。
2. K线形态：长上影、长下影、实体强弱、跳空缺口、突破/假突破、回踩确认、箱体上下沿、前高前低。
3. 量价关系：量比、成交额、换手率、放量上涨、放量滞涨、缩量回踩、下跌放量、尾盘放量/缩量。
4. 技术指标：MACD、KDJ、RSI、BOLL、均线多空排列、背离信号。指标只能辅助判断，不能单独决定买卖。
5. 资金流：主力净流入/净流出、超大单/大单/中单/小单方向、净流入金额、净流入占成交额比例、连续多日资金趋势。
6. 盘口/暗盘资金：盘中主动买卖差、大单承接、撤单/压单/托单、封单/炸板资金、尾盘抢筹或尾盘出货迹象。若数据源没有“暗盘”或盘口细节，必须说明缺失，并用可得的实时资金流、成交额、分时量价和尾盘行为替代。
7. 板块资金：所属板块、ETF、核心成分股或同主线标的的资金流向和强弱对比。
8. 外围资金影响：当涉及科技、AI链、半导体、通信、消费电子时，结合美股期货、纳指、费半、日韩市场、相关龙头股表现。

输出时必须有“资金面/盘口复核”或等价小节，明确写：
- 净流入/净流出金额和方向；
- 是主动承接、分歧换手、拉高出货，还是缩量观望；
- 资金信号与技术信号是否一致；
- 如果资金数据缺失，缺失原因和替代依据。

资金规则优先级：硬风险 > 趋势结构 > 资金持续性 > 短线技术指标。单日资金流不能直接推翻长期趋势，但连续资金流出叠加放量滞涨、跌破关键位时，必须下调动作等级。

## 数据真实性护栏（强制执行）

本 skill 的首要约束：宁可输出“未取得数据/无法确认”，也不能把推测、旧数据、模型常识、截图外信息或网页未验证内容写成事实。

### 1. 数据分级

所有行情、资金、新闻、财报、外围市场、量化信号、盘口/暗盘信息必须按来源分级：

- 实时/最新在线数据：必须写清来源、抓取时间或数据日期。
- 用户截图数据：必须标注“来自用户截图”，只能使用截图中可见字段。
- 历史缓存/旧报告数据：必须标注原始日期，不能冒充今天数据。
- 推断/经验判断：必须写“推断”“可能”“倾向”，不得写成确定事实。
- 缺失数据：必须写“数据缺失/未取得”，并说明缺失原因和替代依据。

### 2. 禁止伪造

以下行为禁止：

- 没有联网核验却写“今日主力流入/流出”“最新公告”“盘中资金”。
- 把上一交易日收盘数据当作当天盘中实时数据。
- 把无法打开的网页、404 的阿布量化页面、空结果写成已验证信号。
- 把截图看不见的成交、成本、数量、条件单或盘口数据补脑出来。
- 用“应该/大概率”包装成事实。
- 资金流、盘口、暗盘、筹码、业绩预期、机构观点缺失时编一个数字。

### 3. 在线核验要求

当问题涉及“今天、现在、开盘、盘中、明天、最新、资金流、外围影响”时，必须先尝试在线核验。至少核验：

- A股/ETF/指数：价格、涨跌幅、成交额、量比、换手率、主力净流、超大单/大单方向、数据时间。
- 技术结构：日线/周线 K 线、BBI、MA5/MA10/MA20/MA60 的计算日期。
- 外围：纳指/纳指100、费半/SOX 或半导体 ETF、美股期货、日韩市场、AI链核心股。
- 新闻/公告：优先官方公告、交易所、公司公告、可靠财经媒体；没有就写缺失。

如果联网失败、接口返回旧日期、页面打不开或数据源互相冲突，必须在报告开头加入：

```text
数据可靠性提示：本报告哪些数据已验证，哪些未验证，哪些来自截图或旧数据。
```

### 4. 交叉验证与冲突处理

关键数字必须尽量交叉验证：

- 行情与资金流优先用公开行情源，并记录源名称。
- 若两个来源不同，报告中写出差异，采用较保守结论。
- 资金流单日数据不能单独决定买卖，必须与量价、BBI、周K、仓位纪律共振。
- 阿布量化、AI策略页若更新时间早于当前交易日，必须标注“滞后辅助信号”。

### 5. 报告必备字段

每份 report、复盘、明日计划、决策仪或盘中应对必须有“数据可靠性”小节，至少包含：

- 数据日期/抓取时间；
- 用户截图信息；
- 在线行情来源；
- 未取得的数据；
- 本报告中哪些结论是事实，哪些是推断。

## Report 正确性合同（强制执行）

生成 report 时必须参考成熟分析报告的结构：先回答问题，再给证据，再给限制，最后给可执行动作。不要把报告写成行情散文，也不要只给情绪安慰。

### 1. Report Spine

每份 report 在动笔前必须先形成以下骨架，最终报告也要能看出这些内容：

- 用户当前问题：今天复盘、明日计划、盘中应对、减仓建议、加仓建议或条件单。
- 决策对象：具体标的、账户仓位、现金、持仓角色。
- 决策答案：买入、加仓、持有、减仓、清仓、观察或禁止操作。
- 证据链：行情/资金/量价/BBI/周K/外围/截图/成交记录。
- 关键不确定性：哪些数据缺失、哪些信号冲突、哪些结论只是推断。
- 可执行计划：触发价、确认时间、动作、数量/比例、失效条件。
- 复盘口径：明天或后续如何判断本报告对错。

### 2. Claim-Evidence-Caveat-Action

报告里的每个核心结论必须同时具备四件事：

| 项目 | 必须回答 |
| --- | --- |
| Claim 结论 | 我现在认为应该做什么？ |
| Evidence 证据 | 这个判断基于哪些数据、截图或来源？ |
| Caveat 限制 | 哪些数据没拿到、信号冲突或可能误判？ |
| Action 动作 | 触发什么条件后，买/卖/持有多少？ |

如果一个结论没有证据，只能写成“观察假设”，不能写成行动建议。  
如果一个行动没有触发条件和失效条件，必须降级为“方向性看法”，不能写成交易计划。

### 3. 数值一致性核对

涉及持仓、成交、仓位、现金、盈亏时，必须做一致性核对：

- 持仓数量 = 截图可见持仓 + 已知成交变化；若无法闭合，写“持仓估算/待确认”。
- 市值约等于数量 × 当前价；差异明显时要标注可能原因。
- 仓位 = 市值 / 总资产；若截图给出仓位，以截图为准并说明时间。
- 卖出/买入后的现金变化要与截图现金大致一致；不一致时不能硬推。
- 成本异常、ETF 成本显示异常、持仓可用数量与总数量不一致时，必须单独提示。

### 4. 置信度与数据质量联动

`confidence_level` 不能只看语气强弱，必须受数据质量约束：

- 数据完整、来源新、价格/资金/持仓能交叉验证：最高可为“高”。
- 有截图但缺少实时资金、外围或关键技术数据：最高为“中”。
- 数据来自旧截图、旧接口、阿布量化滞后或关键价格无法确认：最高为“中低/中”。
- 资金流、持仓数量、当前价格、交易日期任一关键字段缺失：不能给“高”。
- 结论高度依赖推断时，必须在 `confidence_reason` 或报告中写明。

### 5. DecisionSignal 映射

当 report 产生明确动作时，必须能映射到 `DecisionSignal`，便于后验评估：

- `action`：buy/add/hold/reduce/sell/watch/avoid/alert 之一。
- `horizon`：intraday/1d/3d/5d/10d/swing/long。
- `market_phase`：premarket/intraday/lunch_break/closing_auction/postmarket/non_trading/unknown。
- `confidence`：由数据质量、证据一致性、仓位风险共同决定。
- `watch_conditions`：至少 2 条可观察条件，不能只写“看盘面”。
- `invalidation`：明确写出判断失效条件，例如跌破某价且 15 分钟收不回。
- `evidence`：列出行情、资金、BBI、截图、外围、阿布量化等证据摘要。
- `data_quality_summary`：列出已验证、未验证、滞后、缺失的数据。

### 6. 标准 Report 结构

除非用户明确要求极简，否则 report 应按以下顺序组织：

1. 标题：日期 + 用途，例如“2026-07-02 持仓复盘与明日应对报告”。
2. 数据可靠性：数据来源、截图时间、行情时间、未取得数据。
3. 一句话结论：先给答案，不绕。
4. 当前账户与持仓：总资产、仓位、现金、每个标的数量/成本/现价/盈亏/角色。
5. 今日操作复盘：每笔交易是否符合纪律，是否追涨杀跌。
6. 市场与外围：A股指数、板块、资金、日韩/美股/纳指/费半/AI链影响。
7. 持仓逐只分析：技术面、资金面、BBI/周K、相对强弱、风险线。
8. 个人规则复核：买强不买弱、去弱留强、长期趋势票 BBI。
9. 明日或盘中执行计划：开盘 15 分钟、10:30 前后、14:30 后。
10. 条件单/触发器：触发价、确认时间、动作、数量/比例、失效条件。
11. 禁止操作：当前最容易犯的错。
12. 后验检查：明天用什么结果验证今天报告是否正确。

### 7. 交付前自检

输出 report 前必须检查：

- 是否回答了用户真正的问题，而不是只复述行情？
- 是否每个买卖建议都有触发价、确认时间、数量/比例和失效条件？
- 是否区分了事实、截图、在线数据、推断和缺失？
- 是否有至少一处与原始截图/成交记录对应的持仓核对？
- 是否检查了 BBI、周K、相对强弱和仓位纪律？
- 是否把外围影响写成“影响条件”，而不是绝对预测？
- 是否把高置信度建立在真实数据上，而不是语气上？
- 是否明确哪些操作今天/明天禁止？
- 是否能在后续复盘中判断本报告对错？

## 输出结构 (`AnalysisResult`)

分析函数返回一个 `AnalysisResult` 对象（或其列表），该对象具有丰富的结构。以下是其关键组件的简要概述，并附有真实的输出示例：

`dashboard` 属性包含核心分析，分为四个主要部分：
1.  **`core_conclusion`**: 一句话总结、信号类型和仓位建议。
2.  **`data_perspective`**: 技术数据，包括趋势状态、价格位置、量能分析和筹码结构。
3.  **`intelligence`**: 定性信息，如新闻、风险警报和积极催化剂。
4.  **`battle_plan`**: 可操作的策略，包括狙击点（买/卖目标）、仓位策略和风险控制清单。

## 持仓报告补齐规则

## 投资者收益纪律层（强制执行）

本 skill 的目标不是预测每一根 K 线，而是帮助投资者提高长期收益质量：保住本金、控制回撤、减少追涨杀跌、把强主线拿住、把弱仓和情绪仓及时处理掉。

以后凡是生成股票分析、持仓 report、交易复盘、明日计划、条件单方案或“现在怎么办”，必须额外执行以下纪律层。

### 1. 仓位分层

每次分析都要把持仓分为：核心仓、进攻仓、试错仓、观察仓、防守现金。

- 核心仓：主线最强、逻辑最稳、用户愿意承受波动的仓位。
- 进攻仓：趋势强但波动大，只能在市场环境允许时参与。
- 试错仓：验证想法的小仓位，原则上单个不超过总资产 5%-8%。
- 观察仓：只跟踪不买入，除非触发明确条件。
- 防守现金：市场风险升高时必须保留的现金。

仓位纪律：

- 大盘正常：总仓位可维持 70%-90%，但必须保留现金。
- 大盘分歧：总仓位建议 60%-75%，只留强主线。
- 系统性杀跌：总仓位建议 40%-65%，禁止补弱仓。
- 单日账户亏损超过 3%：禁止加仓，只能持有、减仓或清弱仓。
- 单日账户亏损超过 5%：进入防守模式，只允许减仓、清仓、设置条件单。

### 2. 交易前检查清单

任何买入、加仓或追回卖出仓位前，必须回答：

1. 这是买强，还是买便宜？
2. 它是核心主线，还是杂仓试错？
3. 价格距离日线 BBI、周线 BBI 多远？
4. 当前主力资金是流入还是流出？
5. 板块和同主线核心标的是否同步走强？
6. 如果判断错了，跌破哪里卖？
7. 当前账户仓位是否允许加仓？
8. 这笔交易是否会让现金不足？

若以上任一项无法回答，动作必须降级为“观察”或“小额试探”，不能重仓。

### 3. 条件单模板

当用户早上有事、不能盯盘、或要求“明天怎么办”时，必须输出条件单模板：

```text
若 标的 跌破 X 且 N 分钟不能收回：卖出 Y
若 标的 冲高到 X 但量能/资金不跟：卖出 Y
若 标的 回踩 X 后收回并有资金回流：观察或小额试探 Y
若 大盘/板块触发系统性风险：禁止加仓，优先减弱仓
```

条件单必须写清：触发价、确认时间、动作、数量或比例、失效条件。

### 4. 日内三阶段决策

日内建议必须拆成三段：

- 开盘 15 分钟：只看承接，不急着追高或割肉。
- 10:30 前后：判断是真弱、洗盘、还是高开低走派发。
- 14:30 后：决定留仓、防守减仓、或尾盘条件单。

若用户无法盯盘，优先使用条件单和仓位上限，不能依赖临场反应。

### 5. 回撤保护

必须跟踪账户级回撤：

- 当日亏损超过 3%：进入黄灯，停止买入。
- 当日亏损超过 5%：进入红灯，只允许减仓或持有核心仓。
- 从阶段高点回撤超过 8%-10%：进入防守周，减少交易频率。
- 连续两次追涨杀跌复盘失败：下一次买入必须减半。

回撤保护优先级高于看多理由。

### 6. 卖飞处理

用户卖出后，如果标的继续上涨，不允许立刻情绪追回。必须执行：

- 先判断是否重新站回关键价位。
- 再确认资金是否回流。
- 再确认大盘和板块是否修复。
- 买回数量不得超过原卖出数量的 30%-50%，除非出现明确趋势重启。
- 如果只是日内脉冲，不追回。

卖飞不是错误；卖飞后失控追回，才是错误。

### 7. 加仓与减仓优先级

加仓优先级：

1. 强主线核心仓回踩 BBI 后收回支撑。
2. 盘中闪崩后快速收回关键位并出现承接。
3. 板块资金回流，强标的先修复。
4. 试错仓只允许小额，不允许摊低。

减仓优先级：

1. 试错仓、杂仓、弱于主线的仓位。
2. 资金持续流出的持仓。
3. 跌破关键支撑且不能收回的仓位。
4. 高开低走、放量滞涨、拉高出货嫌疑的仓位。

最终输出必须说明：当前动作是为了进攻、保护利润、降低回撤，还是纠错。

## 阿布量化策略信号接入

当用户要求生成 report、明日计划、决策仪、市场复盘，或明确提到 abuquant / 阿布量化 / 量化策略网站时，必须加入“阿布量化策略信号”小节。

默认使用脚本：

```text
scripts/fetch_abuquant_signals.py 000001 399006 515880 601138 300502 --format markdown
```

页面模板：

```text
https://abuquant.com/abu_context/output_gi_week3/report/{code}/pc.html
https://abuquant.com/abu_context/output_gi_week3/report/{code}/pc_mc.html
```

接入内容至少包括：

- 页面更新时间；
- 买入卖出AI信号量化结论；
- 多方/空方评分；
- 趋势跟踪、均值回复、止损风控、止盈利保的激活数量和占比；
- 关键指标摘要：KDJ、MACD、BOLL、RSI、CCI；
- 指标形态，如 KDJ死叉、布林上轨/下轨、看多/看空；
- 与本报告的 BBI、资金流、仓位纪律是否一致。

使用边界：

- 如果脚本失败、页面 404、页面没有对应标的、页面更新时间滞后或字段抓取为空，必须写明失败原因和更新时间，不得生成虚假的多空评分、指标状态或买卖 AI 信号。
- 阿布量化信号是辅助证据，不是直接买卖指令。
- 若阿布量化看多，但资金流大幅流出、仓位过高或 BBI 未收回，动作仍需降级。
- 若阿布量化看空，但标的在周线 BBI 附近出现长下影和资金承接，可列为观察，不可直接清仓。
- 报告必须写清“量化信号与人工规则是否共振”：共振、背离、或数据不足。
